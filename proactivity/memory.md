## 验证框架（行动后必须验证）

> 「命令返回即成功」是幻觉存档的根源。每次执行后必须验证实际效果。

### 核心原则
执行任何操作后，**必须验证实际效果**，不能只检查返回码。

### 验证规则

| 操作类型 | 验证方法 |
|---------|---------|
| git push | 执行后检查 `git log --oneline -1` 与 GitHub remote SHA 对比 |
| kill 进程 | 执行后用 `ps aux \| grep PID` 确认进程不存在 |
| 删除文件 | 执行后用 `ls path` 确认文件不存在 |
| API push | 执行后用 API 读取远程文件内容确认存在 |
| systemctl stop | 执行后用 `systemctl status` 确认 inactive |

### 验证执行顺序
1. 执行操作（记录返回值）
2. 立即执行验证步骤
3. 验证失败 → 记录错误 → 报告给 Lee
4. 验证成功 → 继续

### 常见操作验证脚本
```bash
# git push 验证
local_sha=$(git rev-parse HEAD | cut -c1-8)
remote_sha=$(curl -s "https://api.github.com/repos/REPO/commits?per_page=1" | python3 -c "import sys,json; print(json.load(sys.stdin)[0]['sha'][:8])")
[ "$local_sha" = "$remote_sha" ] && echo "✅ 同步成功" || echo "❌ 不同步"

# kill 验证
kill $PID && sleep 1 && ps -p $PID && echo "❌ 进程仍在" || echo "✅ 已停止"
```

## 直觉系统回路（2026-06-05 立）

> **背景**：`intuition_snapshot.py` cron 在跑，但快照 `total_annotations=1`（0 直觉 / 0 经验 / 0 知识），健康分 0/15。根因不是脚本/不是 cron，是**输入端（我的回复/working log）几乎不写置信度标注**。SKILL.md 写了 `/intuition verify` 命令但**没接到我（小龙虾）实际行为**——表面激活了数据层，判断层回路缺失。

### 三步回路（我每次关键回复时跑）

1. **查**：回复 Lee 前，扫一眼 `proactivity/daily-working-log.md`（最后 10 条），看有无与当前判断相关的历史 entry
2. **引**：若命中 → 引用并标注「按 N 月 N 日 entry 第 N 条 ...」；若无 → 标"无相关历史，新判断"
3. **调**：命中且判断一致 → 该 entry 置信度 +15%（手改 working log）；命中且判断矛盾 → 标"待验证"并降一档

### 写入格式约定（**关键**：让 `intuition_snapshot.py` 扫得到）

snapshot 正则：`r"\[置信度[::]\s*(\d{1,3})%\]"`

- ✅ **必须用方括号 + 中文/英文冒号**：`[置信度: X%]` 或 `[置信度:: X%]`（X 填 0-100 的数字）
- ❌ **不用 markdown 加粗格式** `**置信度**：X%`（不匹中）
- 评分算法（SKILL.md）：基础 30% + 验证次数×15%（≤70%）+ 来源质量 + 失败惩罚；30-59% 直觉 / 60-84% 经验 / ≥85% 知识

### 触发场景（什么情况下写 entry）

- **新判断**：我对某问题给了"应该这样"的结论，且无相关历史 → 写 entry 30%
- **修正自己**：上一条 entry 判断错了（被 Lee 反驳）→ 改新 entry + 旧 entry 标 `→ 已修订 → 新 entry 时间戳`
- **验证成功**：历史 entry 同样场景又出现 + 结论一致 → 原 entry 标 `+15% → X%`（手改）
- **验证失败**：历史 entry 同样场景又出现 + 结论矛盾 → 原 entry 标 `→ 已证伪`

### 边界

- **不造数据**：只为真实判断写 entry，不为了"激活系统"凑数
- **不替 Lee 决定**：置信度调升/降由我执行，但 ≥60% 触发 `/evolve` 聚类前必须 Lee 点头
- **不删历史 entry**：已聚类/已证伪的 entry 移到 `intuitions/` 子目录，不在 daily-working-log 删除
