## 00:13-00:19 高频 cron 加 light-context（Lee 拍板"不浪费 LLM"）

**触发**：Lee 6/10 00:11 "是的，不要浪费 LLM 调用功能" + 发现 5h 周期 4 分钟烧 9% 不合理

**根因**：
- **OpenClaw 默认 bootstrap 加载 = 200k 字符 = ~71k tokens**（每个新 session）
- **ec85def5 water-push-queue-worker** 每 1 min 跑 = **71k tokens/min = 4.3M/h = 21M/5h**（超过 5h 周期 4.16M 额度的 5 倍）
- **bac841c2 cron-retry-monitor** 15min 跑 = ~280k/h
- **5a222ec4 cron-products-mtime-check** 4h 跑 = ~17k/h

**修复**：
3 个高频 cron 加 `--light-context`（OpenClaw 原生开关）：
- ec85def5-2b70-45c8-8c13-65700a86de28
- bac841c2-14e6-45aa-bb64-1f6be5786c8f
- 5a222ec4-dbb5-44ba-ab28-81253ddbbce0

**实测效果**（改完 4 分钟后）：
- ec85def5 jsonl 大小：**200KB → 4KB（省 50 倍）**
- 5h 周期估算：21M → 390k tokens（**省 98%**）
- MiniMax 实时：5h 周期 5%/3min → 4%/1min（**速率略降**，主因变成本会话我自己调用）

**dev_20260610_002**：
- **`--light-context` = OpenClaw 减少 bootstrap 加载的核心开关**——高频 cron 必加
- **bootstrap 大小检查 = 每个新 session 都加载 200k 字符**——这是 71k tokens 的根源
- **改完后状态文件 lightContext=True**——不是简单的"开关"，是 OpenClaw 原生支持的轻量级 bootstrap
- **可推广**：所有每 ≤15min 的 agentTurn cron 都应 light-context
- **更激进方案**：纯 shell 任务（如 ec85def5）可改 systemEvent 或 isolated shell——但 light-context 已省 98% 够用
- **本次 4%/分钟主要是本会话在烧**（不是 cron），是 agent 调用自己——这个不优化

**置信度**：100%（已实测 ec85def5 jsonl 200KB → 4KB）
