### Kimi For Coding 平台限制
- `https://api.kimi.com/coding/v1` **仅供编程 CLI 工具**（Claude Code/Roo Code/Kilo Code/Kimi CLI）
- 普通 Agent（包括 OpenClaw）会被 `access_terminated_error` 拒
- **不要**把 Kimi For Coding 当通用模型接入 Agent 系统

### Anthropic transport 必要参数
- OpenClaw minimax provider 用 anthropic-messages 协议
- 必须显式设 `maxTokens > 0`，否则调用直接 0 字截断
- M2.7 / M3 都需要，但 M3 默认配置里有，M2.7 漏了

### 推理题设计陷阱
- 4 个模型互评 T4 时我设计的"4 球盒子"题**有多个解**，反复改题 3 次
- **教训**：客观题用"找解"型（数字谜题 a+b+c=999）比"推理唯一结论"更可验证

### 互评去自评机制

## 教训（追加）

### 飞书 message 工具 schema 必填项

- **症状**：第一次 `message action=send` 报 `Validation failed: text must have required properties text`
- **根因**：schema 要求 `text` 字段在 message 对象**内部**，不是顶层
- **正确格式**：`{ "action": "send", "channel": "feishu", "target": "oc_xxx", "message": "文本" }`（顶层用 `message` 字段，内部文本）
- **教训**：调 message 工具前先看 schema 例，不要凭直觉拼参数

### Cron state.lastRunAt 全空 ≠ 从未跑

- **症状**：`jobs.json` 里所有 cron `state.lastRunAt` 都是空，`state.lastStatus="never"`
- **结论**：这只是**gateway state 没回写**的表面现象
- **真实状态**要用 `cron runs <id>` 看历史（那个返回真实执行记录）
- **教训**：不要被 jobs.json 的"never"误导；要查实际运行用 `cron runs`

### 推理题 vs 计算题的稳定性

- 4 模型对比 + 双盲测试**唯一稳定的区分度在数学计算**（V4 系列严于 M 系列）
- 创意 / 摘要 / 营销 / 诗**所有模型表现高度一致**（互评分数几乎相同）
- **结论**：评估模型时多用计算题 + 代码题，少用创意题（区分度低、裁判主观空间大）
