## 教训（系统级）

### systemEvent 调 shell 脚本的隐含陷阱

- **错**：以为 `systemEvent` 跑通 = 推送完成
- **对**：`lastDeliveryStatus: not-requested` 是 systemEvent 模式的**正常状态**，**不等于**"消息已送达"
- **验证步骤**：必须看脚本是否自包含推送逻辑（`openclaw message send` / `curl feishu` 等）
- **下次遇到 systemEvent + shell 脚本**：先 `cat` 脚本末尾看是否有发送动作，再判断"成功"是脚本跑完还是消息送达

### 看 cron-retry.log 容易误导

- 早上看到 `cron-retry.log` 里大量 `FailoverError: Token Plan 限流`，**反推**记忆系统崩
- 实际：那些是别的 cron（Self-Improving 周报 / PARA-Inbox / 轨迹挖掘等）的历史错误

- 教训：看到批量失败时**先确认涉及哪些 cron**，不要一句话反推为"全系统崩"
