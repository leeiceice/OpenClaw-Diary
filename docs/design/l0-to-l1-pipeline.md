## 1. 背景

### 现状
- **L0**（每日记忆）= `memory/YYYY-MM-DD.md`，实时写入 + `l0_watchdog.sh` 守护
- **L1**（中期记忆）= 当前**只有叙事型周报** `evolution-report-YYYY-MM-DD.md`（每周日 19:15 自动跑），**缺结构化主题聚类**
- **L2**（长期记忆）= `MEMORY.md`，**手动维护**（Lee 拍板）
- **链路状态**：
  - L0 ✅ 稳定（端到端验证 6/7 通过）
  - L1 ⚠️ 半断（周报有人写，主题聚类没人做）
  - L2 ⚠️ 半断（无候选推荐机制，全靠 Lee 主动提）

### 痛点
- **今天 6/7** 我修了 7 个 `dev_20260607_00X`，但**全散在 `corrections.md` 里**，没有"今天发生了什么主题"的结构化视图
- **周报 evolution-report-2026-06-07.md** 是叙事格式，**不是主题聚类**——读起来费劲，跨周主题断
- **L2 候选提取靠人**——我每次提示"要不要写 MEMORY.md"，**应该是系统主动提**

### 目标
**让 L0 → L1 → L2 三层链路**：
- L0 → L1 **自动化**（每周日 22:00 跑）
- L1 → L2 **半自动**（生成候选，Lee 拍板写入）
- L2 → MEMORY.md **手动**（Lee 拍板决定写不写、写到哪）

## 2. 设计方案

### 2.1 L1 文件结构（B 方案）

```
memory/l1/
├── weekly-2026-W22.md
├── weekly-2026-W23.md  ← 下周日（6/14）会生成
├── weekly-2026-W24.md
└── ...
```

**文件名**：`weekly-YYYY-Www.md`（ISO 周编号，与 evolution-report 内容头的 W23/W24 一致）
**触发日**：每周日 22:00 CST（晚于 evolution-report 19:15，留 3 小时给叙事周报先生成）

### 2.2 L1 文件内容结构

```markdown
# L1 主题聚类 - 2026 W{ww}（YYYY-MM-DD ~ YYYY-MM-DD）

> 自动生成：基于本周 L0 + corrections.md 主题聚类
> 配套周报：evolution-report-YYYY-MM-DD.md
> 配套 L2 候选：见文末「L2 候选指标」

## 主题 1: {主题名}（置信度 X%）

**关联 dev**：
- dev_YYYYMMDD_001
- dev_YYYYMMDD_003

**核心事件**（按时间线）：
- 17:00 触发事件
- 18:33 修复完成
- 19:39 端到端验证

**核心教训**（1-2 句）：
- ...

**永久机制**（如有）：
- 脚本：xxx.sh
- cron：xxx (id)
- 监控：xxx

## 主题 2: ...

## 跨主题模式

- 共同根因：xxx
- 修复方向：xxx

## L2 候选指标（≥85% 可提升到 MEMORY.md）

| 指标 | 当前置信度 | 来源 | 建议写入主题 |
|------|----------|------|------------|
| "沉默失败" 是系统性模式 | 90% | 5/21 5/29 6/2 6/7 4 次实证 | 反信息茧房 / 沉默失败 |
| "chat_id 验证分三层" | 85% | 6/7 3 个 cron 实证 | 推送路由 / 沉默失败 |

---

_生成时间：YYYY-MM-DD HH:MM | by 小龙虾（L0→L1 流水线首次运行）_
```

### 2.3 触发流程

**每周日 22:00 CST** → cron `l0-to-l1-pipeline` 触发：

```
1. 加载本周 L0 文件
   - 读 memory/2026-MM-DD.md（周日及前 6 天，共 7 个文件）
   - 读 self-improving/corrections.md 提取本周 dev 编号
2. 主题聚类（agentTurn + M3 模型）
   - 按 dev_YYYYMMDD_NNN 的主题聚类
   - 提取每个主题的核心事件 + 教训 + 永久机制
3. 生成 L1 文件
   - 写 memory/l1/weekly-2026-Www.md
4. 提取 L2 候选（≥85% 置信度）
   - 从主题中提取"普适原则"（适用于其他场景的规律）
   - 写到文末「L2 候选指标」表格
5. 推送进化群
   - 简报：本周 N 个主题，M 个 L2 候选
   - 附 L1 文件路径
6. 失败告警
   - 失败推进化群
   - cooldown 1h
```

### 2.4 模型与超时

| 阶段 | 模型 | 原因 |
|------|------|------|
| 数据加载（读 L0 + corrections）| 纯 shell | 无需模型 |
| 主题聚类（生成 L1）| **M3（主会话模型）** | 需要深度主题理解 |
| L2 候选置信度评估 | M3 | 需要判断普适性 |
| 失败推送 | 纯 shell | 无需模型 |

**超时**：agentTurn timeout=600s（参考 heartbeat 实际 33s 经验，留 18x 冗余）

### 2.5 L2 候选阈值（85%）

**置信度评估算法**（建议，但待实现时细化）：
- 基础分 = 30%
- + 30% if 多个 dev 实证（≥3 个不同日期提到）
- + 20% if 已有永久机制（脚本/cron 守护）
- + 10% if Lee 拍板确认过
- -20% if 单一来源

**阈值**：
- ≥85% → 写入「L2 候选指标」表（强候选）
- 70-84% → 写入「观察池」（中候选，下次再聚类时看是否升级）
- <70% → 不写入（噪音过滤）

### 2.6 边界与限制

**不做什么**：
- ❌ **不自动写 MEMORY.md**（L2 写入必须 Lee 拍板）
- ❌ **不删除历史 L0**（保留全量）
- ❌ **不修改 L0 文件**（只读）
- ❌ **不与 evolution-report 内容重复**（evolution-report 是叙事，L1 是结构）

**失败处理**：
- L0 缺失某天 → 跳过该天（不阻断）
- corrections.md 格式异常 → 静默跳过（不阻断）
- 模型超时 → failureAlert 推进化群（cooldown 1h）
- 文件写入失败 → 重试 1 次，再失败推群

## 4. 风险与缓解

| 风险 | 概率 | 缓解 |
|------|------|------|
| M3 模型聚类质量差（主题归错）| 中 | 1 周回看 1 次，必要时改 prompt |
| 6/7 7 个 dev 聚类成 3 主题是巧合，下次可能聚成 7 主题 | 中 | 强制 ≤5 主题（prompt 里限）|
| L0 某天缺失导致数据不全 | 低 | watchdog 已建（缺就推群）|
| 候选指标噪音太多（每周都 ≥10 条）| 中 | 阈值 85% 已卡，再不行就 90% |
| cron 撞车（22:00 有别的 cron）| 低 | 提前查 cron list，错开 |
